# 战绩高级筛选 UI 讨论总结

生成时间：2026-03-02

## 讨论范围

本总结覆盖从“当前逻辑是先选这一层用什么 `or / and / not`，再去选条件，这对用户是否友好”开始，到后续关于：

- 高级筛选器的交互顺序
- `相对于其友方成员 / 敌方成员` 的语义
- `以投降失败` 的语义
- “同类取并集，不同类取交集”是否应作为默认逻辑

这几部分讨论。

## 当前实现的本质判断

当前高级筛选器更像一个“AST / 规则树编辑器”，而不是普通用户熟悉的“筛选面板”。

它当前的真实交互模式是：

1. 先决定这一层放什么节点
2. 这个节点可能是逻辑节点：`AND / OR / NOT`
3. 也可能是业务条件节点：如队列、胜负、玩家、英雄
4. 如果先选了逻辑节点，就还要继续往里面塞真正的条件

这套流程对实现很自然，因为它直接映射到底层规则树结构；但对用户不自然，因为用户脑子里先想的是“我要筛什么”，不是“我要先建什么逻辑容器”。

## 关于“先选 AND / OR / NOT，再选条件”是否友好

结论：**对普通用户不友好。**

原因有几个：

- 用户做了两步操作后，可能还没有表达任何实际筛选意图。
- 用户被迫先想“逻辑结构”，再想“业务条件”。
- `NOT` 以空容器出现，很像工程内部结构，不像用户语言。
- 当前编辑过程允许“半成品规则”长期存在，进一步强化了“这是在搭树，不是在筛选”的感觉。

典型反例：

```text
满足其一
  不满足
    [空]
```

这在 AST 角度合法，但在人脑里几乎没有可读性。

## 更符合用户直觉的顺序

更合理的顺序应该是：

1. 先选“具体条件”
2. 再调整条件之间的关系
3. 只有在需要更复杂表达时，才显式创建“分组”

也就是说：

- 默认根层就是“同时满足”
- 用户先加条件卡片
- 条件之间再切换 `并且 / 或者`
- `NOT` 应该是条件或条件组上的“取反/排除”操作，不应该先建一个空 `NOT` 容器
- 显式分组应该是高级动作，不应成为首要入口

## 推荐的默认筛选逻辑

结论：**“同类取并集，不同类取交集”是最常见、也最符合直觉的默认逻辑。**

表达成规则就是：

- 同一类多选条件默认 `OR`
- 不同类筛选块之间默认 `AND`

例子：

```text
(英雄 = 亚索 OR 劫)
AND
(队列 = 单双排)
AND
(时间 = 最近 7 天)
```

这比一开始就让用户手动决定整个树的 `AND / OR / NOT` 更符合一般筛选器的心智模型。

### 需要补充说明的例外

并不是所有“同类”都天然应该 `OR`。

例如“对局包含玩家”这一类，用户有时想表达：

- 包含 A 或 B

有时想表达：

- 同时包含 A 和 B

因此更合理的产品策略是：

- 默认使用“同类并集、不同类交集”
- 对少数歧义大的类目，额外提供“任一满足 / 全部满足”切换

## 关于 `相对于其友方成员 / 敌方成员` 的讨论结论

当前文案不够人话，容易让用户误解。

这里的“其”指的是下拉框中选中的那个人，不是当前页面主角。

所以：

- `对于其友方成员`
  = 以该玩家为参照，取这个玩家所在队伍
- `对于其敌方成员`
  = 以该玩家为参照，取这个玩家的对面队伍

这两个概念本身在逻辑上是正确的，但文案对普通用户不友好。

更易懂的替代表达建议：

- `检查该玩家所在队伍`
- `检查该玩家对面队伍`

或者：

- `以该玩家为参照：他的队友`
- `以该玩家为参照：他的对手`

## 关于 `以投降失败` 的讨论结论

当前 UI 文案容易让用户误以为它是“附加条件”，但实现实际上是“精确匹配一个布尔值”。

当前实际语义是：

- 勾选 `以投降失败`
  = 只匹配投降失败
- 不勾选
  = 只匹配非投降失败

它不是：

- 勾选 = 仅投降失败
- 不勾选 = 所有失败

因此这个控件当前很容易误导用户。

更合理的设计应该改成三态：

1. 所有失败
2. 仅投降失败
3. 仅非投降失败

## 产品判断：高级筛选应该如何定位

当前高级筛选器的底层能力是有价值的，但不应该直接作为普通用户的默认筛选入口。

更合理的分层是：

- 默认给普通用户一个简单筛选器
- 高级用户在需要时进入规则编辑器
- 底层继续统一使用当前规则树 / predicate 引擎

也就是说：

- **简单 UI 可以保留**
- **强内核可以继续用**
- **默认不要直接暴露 AST 编辑思维**

## 推荐的交互方向

### 默认模式

- 采用简单筛选 UI
- 同类多选默认并集
- 不同类默认交集
- 提供少量必要的“任一 / 全部”切换

### 高级模式

- 保留显式逻辑分组
- 允许 `AND / OR / NOT`
- 允许范围切换，如玩家、队友、敌方
- 但要补一层“人话摘要”，让用户能随时读懂当前规则

## 对当前版本的总体评价

结论可以概括为一句话：

**当前实现更像工程师规则编辑器，不像普通用户筛选器。**

具体来说：

- 内核设计是对的
- 表达能力很强
- 但当前 UI 把内部结构过早暴露给用户
- 默认交互顺序与用户思考顺序相反
- 文案层面也有明显“实现语言”痕迹

## 一句话总结

推荐的整体方向是：

**默认按“同类并集、异类交集”的简单筛选器做；高级模式再暴露规则树和显式逻辑分组；`NOT` 作为修饰操作而不是先决容器。**
